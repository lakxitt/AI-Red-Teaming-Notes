# 📘 Retrieval-Augmented Generation (RAG)

---

## Definition

**Retrieval-Augmented Generation (RAG)** is an AI architecture that combines a **Large Language Model (LLM)** with an external knowledge source to generate more accurate, up-to-date, and context-aware responses.

Instead of relying only on the knowledge learned during training, a RAG system first retrieves relevant documents and then uses them as context to generate an answer.

### Formula

```text
RAG = Retrieval + LLM + Generation
```

---

# Why RAG is Needed

LLMs have several limitations:

- Fixed knowledge cutoff
- Cannot access private documents by default
- May generate hallucinations
- Cannot automatically learn newly published information

RAG overcomes these limitations by retrieving information from external sources before generating a response.

---

# RAG Architecture

```text
               User Query
                    │
                    ▼
            Embedding Model
                    │
                    ▼
            Vector Database
                    │
                    ▼
      Retrieve Relevant Documents
                    │
                    ▼
                  LLM
                    │
                    ▼
             Final Response
```

---

# Working of RAG

## Step 1

User enters a query.

---

## Step 2

The query is converted into an embedding (vector).

---

## Step 3

The vector database searches for semantically similar documents.

---

## Step 4

The most relevant documents (**Top-K**) are retrieved.

---

## Step 5

The retrieved documents and user query are provided to the LLM.

---

## Step 6

The LLM generates the final response based on the retrieved context.

---

# Components of RAG

A RAG system consists of the following components:

- User Query
- Embedding Model
- Vector Database
- Retriever
- Large Language Model (LLM)
- Response Generator

---

# Tokens

## Definition

A **token** is the basic unit of text processed by an LLM. A token can be a word, part of a word, punctuation mark, or symbol.

---

## Types of Tokens

### Input Tokens

Tokens sent to the model.

---

### Output Tokens

Tokens generated by the model.

---

### Context Window

The maximum number of tokens an LLM can process in a single request, including:

- System Prompt
- User Prompt
- Retrieved Context
- Conversation History
- Model Response

---

## Why Tokens Matter

- More tokens increase computational cost.
- More tokens increase API cost.
- Exceeding the context window causes information to be ignored or truncated.

---

# Embeddings

## Definition

An **embedding** is a numerical vector representation of text that captures its semantic meaning.

### Example

```text
"Cybersecurity"

↓

[0.31, -0.72, 0.95, ...]
```

---

## Why Embeddings are Used

- Convert text into numbers.
- Capture semantic meaning.
- Enable similarity search.
- Power RAG systems.

---

# Vector Databases

## Definition

A **Vector Database** stores embeddings and retrieves the most semantically similar vectors instead of performing keyword-based searches.

---

## Working

1. Convert documents into embeddings.
2. Store embeddings in the vector database.
3. Convert the user query into an embedding.
4. Compare vector similarity.
5. Return the Top-K most similar documents.

---

## Popular Vector Databases

- FAISS
- ChromaDB
- Pinecone
- Milvus

---

# Hallucination

## Definition

A **hallucination** occurs when an LLM generates incorrect or fabricated information while presenting it confidently.
